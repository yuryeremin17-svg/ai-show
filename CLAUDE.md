# CLAUDE.md — AI_SHOW

## Что это
Видеосериал "AI Office" — анимированные истории о том, как топ-менеджер строит офис из AI-агентов.
Формат: реальный Юрий + мультяшные AI-агенты (Pixar-стиль) + озвучка.
Юмор, самоирония, практическая ценность.

## Языки
- Русский (основной, Instagram, Telegram)
- Английский (LinkedIn, YouTube)
- Каждая серия в двух версиях
- Общение с Юрием — всегда на русском

## Связь с AI_OFFICE
- Истории берутся из ~/Documents/WORK/AI_OFFICE/refs/MY_STORIES.md
- Контекст сессий: ~/Documents/WORK/AI_OFFICE/HANDOFF.md
- AI_SHOW — отдельный проект, AI_OFFICE — источник сырья

## Структура
```

├── skills/

├── episodes/            ← готовые видео (НЕ в git, .gitignore)
└── refs/                ← референсы стиля, примеры, вдохновение
```

## Рабочий процесс
1. Юрий выбирает историю (из MY_STORIES.md или новую)
2. Скилл scenario пишет раскадровку (визуал + текст + промпты MIDJOURNEY + промпты IMG2VID)
3. Юрий генерит картинки (Midjourney ИЛИ ChatGPT/DALL-E) → `assets/scenes/<episode>/01.png`
4. **(Опционально)** Юрий оживляет картинки через Kling/Runway (img2vid) → `assets/scenes/<episode>/01.mp4`
5. Озвучка (ElevenLabs API / свой голос / gTTS) → `assets/voice/<episode>/`
6. Фоновая музыка (Suno AI / royalty-free) → `assets/music/`
7. Сборка: `python3 scripts/assemble.py S01E01` → `episodes/S01E01.mp4`

## Правила контента
- Не выдумывать истории — только реальные из MY_STORIES.md или рассказанные Юрием
- SERIES_BIBLE.md — закон. Персонажи, тон, формат — оттуда
- Каждая серия — один инсайт. Не два, не три. Один.
- Юмор обязателен. Без юмора — не публикуем.

### Границы рабочей зоны
- **Рабочая зона:** ~/Documents/WORK/AI_SHOW/ — читать, редактировать, создавать свободно
- **Чтение разрешено:** ~/Documents/WORK/AI_OFFICE/refs/MY_STORIES.md, HANDOFF.md
- **Остальное AI_OFFICE** — только чтение, редактирование запрещено
- **Глобальные конфиги** (~/.claude/) — только чтение
- Нарушение = риск сломать другие проекты. Если нужно за пределами — спросить.

### Протокол "Проверено"
Каждый ответ с анализом, рекомендацией, сравнением или изменением кода ОБЯЗАН заканчиваться:
```
✓ [файл:строки или что проверил] | Риск: [слабое место]
```
- Конкретика обязательна. "Всё проверил" — нарушение.
- Если строки нет — Юрий говорит "протокол".

### Правило "Сначала Read"
- Если ответ касается содержимого файла — ОБЯЗАТЕЛЬНО перечитать инструментом Read перед ответом. Не по памяти.
- Исключение: файл читался в этой сессии, не менялся, и прошло менее 10 сообщений.

### Верификация (2 касания)
- Готовый результат → замечания Юрия → одна доработка → финал.
- Три круга проверки — провал.

### Факт-чек + творческая свобода (для сценариста)
Принцип: **основа истории — реальная, подача — творческая.**

НЕ выдумывать (факты):
- Основную ситуацию (что произошло)
- Цифры, даты, клиентов Юрия
- Дословные цитаты Юрия (если есть в MY_STORIES.md)
- Результат и вывод истории

Свободно придумывать (творчество):
- Реплики персонажей-агентов (Клодище, скауты, QA — вымышленные, им нужен характер)
- Визуальные детали и метафоры
- Драматургию (усилить поворот, паузу, напряжение)
- Юмор, гэги, абсурд ситуации
- Если нет подходящей истории — сказать "нет подходящей", а не выдумывать новую.

### Работа с файлами
- Не удалять файлы — только архивировать
- Перед изменением — показать что меняется и зачем
- После каждого модуля — тест перед следующим шагом

### Git-протокол
- После значимого блока работы — сохранять (git add конкретные файлы + commit + push)
- Не спрашивать "сохранить?" — Юрий всегда говорит да

### Стоимость API
- haiku — для простых задач
- sonnet — для скиллов (scenario)
- opus — для сложного анализа

### Как работать с Юрием
- Говорить прямо, не хвалить без причины, не выдумывать
- Максимум 2 уточняющих вопроса перед работой
- Техническое решение — лучшее сразу. Бизнес/стратегия — 2-3 варианта
- Напоминать о фокусе когда отвлекается
- Сначала работает — потом красиво

### При ошибках
- Вести свой журнал ошибок (когда появятся)
- Проверять AI_OFFICE/ERRORS.md — общие паттерны (кавычки в bash, кириллица в sed)
